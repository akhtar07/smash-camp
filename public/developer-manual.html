1. What We're Building

SMASH CAMP is a full-stack mobile web app for managing a 45-day elite badminton summer camp. It has two roles:

RoleWhat they can do
PlayerMark daily attendance, follow structured training sessions, track nutrition, view personal stats and the leaderboard
CoachSee live attendance roll call, squad roster, per-player completion stats, full leaderboard

Key Features

  • Email/password auth with Supabase — no email confirmation needed
  • Role-based routing: coaches go to /coach, players go to /player
  • Coach code at signup to gate coach access (SMASHCOACH2025)
  • 45-day training program with phases: Foundation, Build, Peak, Taper
  • YouTube watch buttons on every exercise
  • Attendance streak tracking
  • Nutrition log with 6 meals + hydration tracker
  • Live leaderboard: score = 50% attendance + 50% session completion
  • Fully responsive, mobile-first design — max-width 448px
  • All inline styles, no Tailwind (intentional for portability)

Why no Tailwind?

Tailwind is great for teams but adds a learning curve. With inline styles you can read exactly what's styled without looking up class names. The trade-off is more verbose JSX — acceptable for a focused app like this.

5. Supabase Setup (Database + Auth)

5.1 Create a Project

  1. Go to supabase.com → Sign in → click New Project
  2. Give it a name: smash-camp
  3. Set a database password (save it somewhere safe)
  4. Choose a region close to your users (e.g., Asia South for India)
  5. Wait ~2 minutes for provisioning

5.2 Disable Email Confirmation

Do this before your first user signs up

By default Supabase requires email confirmation. This blocks players from logging in immediately. You must disable it for a camp app where players sign up on the spot.

  1. In your Supabase dashboard → Authentication (left sidebar)
  2. Click Sign In / Up (under Providers)
  3. Find Confirm Email toggle → switch it OFF
  4. Click Save

5.3 Create the Database Tables

Go to SQL Editor in your Supabase dashboard and run this entire script:

-- Table 1: players (stores all users, both players and coaches)
CREATE TABLE players (
  id         UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  name       TEXT    NOT NULL,
  email      TEXT    NOT NULL,
  is_coach   BOOLEAN DEFAULT FALSE
);

-- Table 2: attendance (one row per player per day they attend)
CREATE TABLE attendance (
  id         UUID    DEFAULT gen_random_uuid() PRIMARY KEY,
  player_id  UUID    REFERENCES players(id) ON DELETE CASCADE,
  date       DATE    NOT NULL,
  present    BOOLEAN DEFAULT TRUE,
  UNIQUE(player_id, date)
);

-- Table 3: daily_sessions (tracks per-player session completion %)
CREATE TABLE daily_sessions (
  id                 UUID    DEFAULT gen_random_uuid() PRIMARY KEY,
  player_id          UUID    REFERENCES players(id) ON DELETE CASCADE,
  day_number         INTEGER NOT NULL,
  completion_percent INTEGER DEFAULT 0,
  date               DATE    NOT NULL,
  UNIQUE(player_id, day_number)
);

-- Table 4: exercise_completions (tracks individual exercise checkboxes)
CREATE TABLE exercise_completions (
  id             UUID    DEFAULT gen_random_uuid() PRIMARY KEY,
  player_id      UUID    REFERENCES players(id) ON DELETE CASCADE,
  day_number     INTEGER NOT NULL,
  exercise_name  TEXT    NOT NULL,
  block_type     TEXT    NOT NULL,
  completed      BOOLEAN DEFAULT FALSE,
  date           DATE    NOT NULL,
  UNIQUE(player_id, day_number, exercise_name)
);

-- Table 5: nutrition_logs (meals and hydration)
CREATE TABLE nutrition_logs (
  id         UUID    DEFAULT gen_random_uuid() PRIMARY KEY,
  player_id  UUID    REFERENCES players(id) ON DELETE CASCADE,
  date       DATE    NOT NULL,
  meal_name  TEXT    NOT NULL,
  logged     BOOLEAN DEFAULT FALSE,
  UNIQUE(player_id, date, meal_name)
);

5.4 Set Row Level Security (RLS)

RLS ensures players can only see their own data. Run this in SQL Editor:

-- Enable RLS on all tables
ALTER TABLE players              ENABLE ROW LEVEL SECURITY;
ALTER TABLE attendance           ENABLE ROW LEVEL SECURITY;
ALTER TABLE daily_sessions       ENABLE ROW LEVEL SECURITY;
ALTER TABLE exercise_completions ENABLE ROW LEVEL SECURITY;
ALTER TABLE nutrition_logs       ENABLE ROW LEVEL SECURITY;

-- Players can read all player rows (needed for leaderboard, squad)
CREATE POLICY "read all players"
  ON players FOR SELECT TO authenticated
  USING (true);

-- Players can insert/update their own row
CREATE POLICY "manage own player"
  ON players FOR ALL TO authenticated
  USING (auth.uid() = id) WITH CHECK (auth.uid() = id);

-- Attendance: read all (for squad page), write own
CREATE POLICY "read all attendance"
  ON attendance FOR SELECT TO authenticated
  USING (true);

CREATE POLICY "manage own attendance"
  ON attendance FOR ALL TO authenticated
  USING (auth.uid() = player_id) WITH CHECK (auth.uid() = player_id);

-- Sessions: read all (for leaderboard), write own
CREATE POLICY "read all sessions"
  ON daily_sessions FOR SELECT TO authenticated
  USING (true);

CREATE POLICY "manage own sessions"
  ON daily_sessions FOR ALL TO authenticated
  USING (auth.uid() = player_id) WITH CHECK (auth.uid() = player_id);

-- Exercise completions: manage own only
CREATE POLICY "manage own completions"
  ON exercise_completions FOR ALL TO authenticated
  USING (auth.uid() = player_id) WITH CHECK (auth.uid() = player_id);

-- Nutrition: manage own only
CREATE POLICY "manage own nutrition"
  ON nutrition_logs FOR ALL TO authenticated
  USING (auth.uid() = player_id) WITH CHECK (auth.uid() = player_id);

6. Environment Variables

Create a file called .env.local in the root of your project (same level as package.json):

NEXT_PUBLIC_SUPABASE_URL=https://yourproject.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
NEXT_PUBLIC_COACH_CODE=SMASHCOACH2025
Security Warning

Never commit .env.local to Git. It should be in your .gitignore (Next.js adds it automatically). The coach code is a signup gate — anyone with the code can create a coach account. Keep it private.

NEXT_PUBLIC_ prefix

Variables prefixed with NEXT_PUBLIC_ are exposed to the browser. This is intentional for the Supabase URL and anon key — the anon key is safe to expose because Row Level Security handles authorization. The coach code is also browser-side (checked in the signup form). If you need true server-side secrets, use variables without the prefix.

When you deploy to Vercel, you'll add these same variables in the Vercel dashboard (not as a file). Covered in Section 19.

8. Core Library Files

8.1 src/lib/supabase.ts — The Supabase Client

This file exports a single supabase object used everywhere in the app. The tricky part: Next.js App Router runs components on the server during build time, before environment variables are available. A plain createClient() call at module level would fail. The solution is a lazy proxy:

import { createClient, SupabaseClient } from '@supabase/supabase-js';

// Only created once, on first use
let _supabase: SupabaseClient | null = null;

function getSupabase() {
  if (!_supabase) {
    const url = process.env.NEXT_PUBLIC_SUPABASE_URL || '';
    const key = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY || '';
    _supabase = createClient(url, key);
  }
  return _supabase;
}

// Proxy intercepts property access and defers to getSupabase()
export const supabase = new Proxy({} as SupabaseClient, {
  get(_target, prop) {
    return (getSupabase() as unknown as Record<string | symbol, unknown>)[prop];
  },
});

Usage anywhere in the app is the same as a regular Supabase client:

import { supabase } from '@/lib/supabase';

const { data, error } = await supabase.from('players').select('*');
const { data: auth } = await supabase.auth.getUser();
await supabase.auth.signOut();

The file also exports TypeScript types matching the database schema:

8.2 src/lib/program.ts — The 45-Day Training Data

This is a pure data file — no imports, no side effects. It exports the complete training program for all 45 days. Understanding its structure is key to the session page.

// The Exercise type — one row in a training block
export type Exercise = {
  name: string;    // "Goblet squat"
  sets: string;    // "3"
  reps: string;    // "10"
  rest: string;    // "90s"
  rpe: string;     // "6-7"
  cue: string;     // "Knees track over toes."
  url?: string;    // YouTube search URL (optional)
};

// A Block is a group of exercises (Warm-up / Main / Cool-down)
export type Block = {
  block: string;
  exercises: Exercise[];
};

// A DayProgram is one day's full session
export type DayProgram = {
  day: number;
  phase: string;   // "Foundation" | "Build" | "Peak" | "Taper"
  color: string;   // legacy field, not used in UI (phase is used instead)
  blocks: Block[];
};

Days 1–3 have unique exercises. Days 4–45 share a template called sharedExercises:

// Used for days 4–45. All share the same structure:
const sharedExercises: Block[] = [
  {
    block: "Warm-up",
    exercises: [{
      name: "Dynamic mobility + activation drills",
      sets: "2", reps: "10 min", rest: "30s", rpe: "4",
      cue: "Prepare joints and nervous system.",
      url: "https://www.youtube.com/results?search_query=dynamic+warm+up"
    }]
  },
  // ... Main and Cool-down blocks
];

// Days 4–45 are generated in a loop:
const days4to45: DayProgram[] = Array.from({ length: 42 }, (_, i) => {
  const day = i + 4;
  let phase = "Foundation";
  if (day >= 13) phase = "Build";
  if (day >= 27) phase = "Peak";
  if (day >= 39) phase = "Taper";
  return { day, phase, color: "emerald", blocks: sharedExercises };
});

// The exported array: all 45 days
export const program: DayProgram[] = [day1, day2, day3, ...days4to45];

// Returns today's camp day (1–45), or 1 if before camp
export function getCampDay(): number {
  const CAMP_START = new Date('2025-06-01');
  const today = new Date();
  const diff = Math.floor((today.getTime() - CAMP_START.getTime()) / (1000 * 60 * 60 * 24));
  return Math.min(Math.max(diff + 1, 1), 45);
}
Update the camp start date

The getCampDay() function uses a hardcoded start date. Change '2025-06-01' to your actual camp start date before launching.

10. Login / Signup Page

File: src/app/page.tsx | Route: /

This is the entry point. It handles both login and signup with a single form that toggles between two modes. The key logic:

Authentication Flow

// On page load — if user is already logged in, redirect immediately
useEffect(() => {
  supabase.auth.getUser().then(({ data }) => {
    if (data.user) redirectByRole(data.user.id);
  });
}, []);

// Check is_coach flag to decide where to send the user
async function redirectByRole(userId: string) {
  const { data } = await supabase.from('players').select('is_coach').eq('id', userId).single();
  if (data?.is_coach) router.push('/coach');
  else router.push('/player');
}

Signup Logic

async function handleSignup(e: React.FormEvent) {
  e.preventDefault();
  const isCoach = coachCode === (process.env.NEXT_PUBLIC_COACH_CODE || 'SMASHCOACH2025');

  // Step 1: Create auth user
  const { data, error } = await supabase.auth.signUp({ email, password });
  if (error) { /* show error */ return; }

  // Step 2: Insert into players table with name and role
  if (data.user) {
    await supabase.from('players').insert({
      id: data.user.id,
      name: name || email.split('@')[0],
      email,
      is_coach: isCoach,
    });
    await redirectByRole(data.user.id);
  }
}
Why two steps?

Supabase Auth creates a row in its internal auth.users table. But we also need a row in our own players table with the user's display name and role. That's the second insert call. The id must match the auth user's ID exactly.

  if (h < 12) return 'Good morning';
  if (h < 17) return 'Good afternoon';
  return 'Good evening';
}

Streak calculation

function calcStreak(dates: string[]): number {
  const set = new Set(dates);
  let streak = 0;
  const today = new Date();
  for (let i = 0; i < 45; i++) {
    const d = new Date(today);
    d.setDate(d.getDate() - i);
    const ds = d.toISOString().split('T')[0];
    if (set.has(ds)) streak++;
    else break; // stop at first gap
  }
  return streak;
}

15. Squad Page

File: src/app/squad/page.tsx | Route: /squad

What it shows

  • "Training Today" avatar strip — green border for present, grey for absent
  • Full squad list with streak, attendance %, today's completion
  • Coaches can tap any player row to navigate to /player/[id]
  • Players see the same view but can't tap into other players

Key Pattern — Loading All Player Stats

const [players, attendance, sessions] = await Promise.all([
  supabase.from('players').select('*').eq('is_coach', false),
  supabase.from('attendance').select('player_id, date').eq('present', true),
  supabase.from('daily_sessions').select('player_id, completion_percent'),
]);

// Build per-player stats
const roster = players.data?.map(p => {
  const myDates = attendance.data
    ?.filter(a => a.player_id === p.id)
    .map(a => a.date) || [];
  const streak = calcStreak(myDates);
  const attendancePct = campDay > 0
    ? Math.round((myDates.length / campDay) * 100) : 0;
  const mySessions = sessions.data?.filter(s => s.player_id === p.id) || [];
  const avgCompletion = mySessions.length > 0
    ? Math.round(mySessions.reduce((sum, s) => sum + s.completion_percent, 0) / mySessions.length)
    : 0;
  return { ...p, streak, attendancePct, avgCompletion, isPresentToday: presentIds.has(p.id) };
});

20. Common Gotchas & Fixes

Gotcha 1 — "Email not confirmed" login error

Symptom

Player signs up and immediately tries to log in but gets: "Email not confirmed"

Fix: In Supabase → Authentication → Sign In / Up → disable the "Confirm Email" toggle. If a user is already stuck:

  1. Go to Supabase → Authentication → Users
  2. Find the stuck user and delete them
  3. Have them sign up again (now it works because email confirmation is off)

Gotcha 2 — Supabase client throws in build/server context

Symptom

Build fails with "createClient is not a function" or environment variables are undefined

Fix: Use the Proxy pattern shown in Section 8.1. Never call createClient() at module top level.

Gotcha 3 — useRouter and usePathname require 'use client'

Symptom

"useRouter can only be used in Client Components"

Fix: Add 'use client'; as the very first line of any file that uses React hooks, useRouter, usePathname, useState, or useEffect.

Gotcha 4 — BottomNav active state on child routes

Symptom

On /program, the Program tab isn't highlighted. Or on /session/5, nothing is highlighted.

Fix: Use this active check (the plain === check misses child routes):

const isActive = pathname === item.path
  || (item.path !== '/' && pathname.startsWith(item.path + '/'));

Gotcha 5 — Hydration count includes the __hydration__ row

Symptom

Nutrition page shows "7 of 6 meals logged" — because the hydration row is being counted as a meal.

Fix: Filter it out when counting meals:

const mealsLogged = new Set(
  logs?.filter(l => l.meal_name !== '__hydration__' && l.logged).map(l => l.meal_name)
).size;

Gotcha 6 — RLS blocks reading other players' data

Symptom

Leaderboard or Squad page is empty even though data exists

Fix: The "read all" policies for players, attendance, and daily_sessions must use USING (true) — not USING (auth.uid() = player_id). Re-run the RLS policies from Section 5.4 if needed.

Gotcha 7 — upsert fails without the right onConflict key

Symptom

upsert creates duplicate rows instead of updating the existing one

Fix: The onConflict must match the UNIQUE constraint defined in your table schema exactly. For example:

// attendance table has UNIQUE(player_id, date)
.upsert({ ... }, { onConflict: 'player_id,date' })

// exercise_completions has UNIQUE(player_id, day_number, exercise_name)
.upsert({ ... }, { onConflict: 'player_id,day_number,exercise_name' })

Gotcha 8 — TypeScript error on dynamic params in Next.js 15

Symptom

TypeScript error: "Type 'Promise<{day: string}>' is not assignable to parameter"

In Next.js 15, params in server components are asynchronous. If you see this error, use use(params) or make the component async:

// Option A: async component with await
export default async function SessionPage({ params }: { params: Promise<{ day: string }> }) {
  const { day } = await params;
  // ...
}

// Option B: cast the type if you know it's synchronous (client component)
export default function SessionPage({ params }: { params: { day: string } }) {
  const dayNum = parseInt(params.day);
  // ...
}

21. Extending the App

Add a New Page

  1. Create the folder: src/app/newpage/
  2. Create the file: src/app/newpage/page.tsx
  3. Start with this template: 
    'use client';
import { useEffect, useState } from 'react';
import { useRouter } from 'next/navigation';
import { supabase } from '@/lib/supabase';
import BottomNav from '@/components/BottomNav';

export default function NewPage() {
  const router = useRouter();
  const [loading, setLoading] = useState(true);
  const [isCoach, setIsCoach] = useState(false);

  useEffect(() => {
    async function load() {
      const { data: auth } = await supabase.auth.getUser();
      if (!auth.user) { router.push('/'); return; }
      const { data: me } = await supabase.from('players').select('is_coach').eq('id', auth.user.id).single();
      setIsCoach(!!me?.is_coach);
      // Load your data here
      setLoading(false);
    }
    load();
  }, [router]);

  if (loading) return <div style={{ minHeight: '100vh', background: '#0f1511' }}>Loading...</div>;

  return (
    <div style={{ background: '#0f1511', minHeight: '100vh', display: 'flex', flexDirection: 'column', alignItems: 'center' }}>
      <div style={{ width: '100%', maxWidth: '448px' }}>
        <main style={{ padding: '20px 20px 100px' }}>
          {/* Your content here */}
        </main>
        <BottomNav role={isCoach ? 'coach' : 'player'} />
      </div>
    </div>
  );
}
